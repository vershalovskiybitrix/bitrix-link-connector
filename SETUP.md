# Инструкция по сборке, установке и публикации

## Как устроен деплой (важно понять один раз)

Приложение — **«Статичное»**: один `index.html`, упакованный в `app.zip` и загружаемый
напрямую в Битрикс24. Отдельный хостинг (GitHub Pages и т.п.) **не нужен** — Битрикс сам
раздаёт файл со своего CDN каждому порталу, который установил приложение.

Это работает благодаря динамическому определению адреса обработчика прямо в коде:
```javascript
var HANDLER_URL = window.location.href.split('?')[0];
```
Какой бы CDN-адрес Битрикс ни выдал конкретному порталу — приложение само его подставит
в `imconnector.register` и `placement.bind`. **Хардкодить и менять `HANDLER_URL` вручную
не требуется** — это отличие от более ранних подходов, где адрес приходилось прописывать
руками после деплоя на внешний хостинг.

## Шаг 1. Сборка `app.zip`

После любой правки `index.html` или `mail.html`:
```powershell
Compress-Archive -Path 'd:\Progs\BitrixPlugin\index.html','d:\Progs\BitrixPlugin\mail.html' -DestinationPath 'd:\Progs\BitrixPlugin\app.zip' -Force
```

## Шаг 2. Создать/обновить приложение в Битрикс24

**Настройки → Разработчикам → Другое → Создать приложение** (или раздел локальных/статичных
приложений — название пункта меняется между версиями интерфейса, актуальную навигацию
смотреть на `helpdesk.bitrix24.ru`, не в обучающих курсах — там устаревшие скриншоты).

- Тип: **Статичное** (Simple Application)
- Загрузить файл: `app.zip`
- Название: «Свои ссылки на мессенджеры»
- Права (scopes) — по фактическим вызовам в `index.html`. Важно: отдельного scope
  `imconnector` не существует — методы `imconnector.*` идут под правом `imopenlines`
  (подтверждено документацией метода `imconnector.register`: `Scope: imopenlines`).
  Реальный набор:
  `im`, `imopenlines`, `placement`, `contact_center`

> ⚠️ При пересоздании приложения после удаления старого Битрикс может не сразу принять
> новую загрузку — если «не даёт перезагрузить», подождать или попробовать другое имя/код.

## Шаг 3. Первый запуск — самонастройка

Открыть приложение один раз (по плитке/ссылке `.../marketplace/app/[ID]/`).
В `BX24.init` приложение само:
1. вызывает `placement.bind({PLACEMENT:'CONTACT_CENTER', ...})` — создаёт плитку
   «Свои ссылки на мессенджеры» в Контакт-центре (раздел «Партнёрские решения» внизу страницы);
2. подгружает сохранённый список включённых мессенджеров (`app.option.get`) и
   регистрирует их коннекторы (`imconnector.register`).

Установочного хука у статичных приложений нет — поэтому плитка появляется только
**после первого открытия приложения человеком**, не сразу после установки.

## Шаг 4. Сквозная проверка (полный цикл)

1. Открыть приложение → экран настроек/чек-листа → «Активировать».
2. Контакт-центр → плитка «Свои ссылки на мессенджеры» → выбрать мессенджер,
   ввести ссылку, выбрать Открытую линию → подключить.
   (или: Открытые линии → линия → коннектор → настройка прямо там же).
3. CRM → Виджет на сайт (`/crm/button/edit/N`) → добавить эту Открытую линию.
4. Открыть страницу с виджетом на реальном сайте, кликнуть кнопку коннектора —
   должен происходить **прямой переход по ссылке** (не открытие чата).

### Как проверить шаг 4 не руками, а само

Статический фетч (curl/WebFetch) не видит виджет — он рисуется JS уже в браузере.
Рабочий способ — headless-браузер (Playwright уже установлен в Python на этой машине,
см. скил `browse-web`):
```python
# открыть сайт → кликнуть по кнопке виджета → page.evaluate() вытащить реальные
# href и классы кнопок (ui-icon-service-clnk_telegram и т.п.)
```
Так было подтверждено вживую на `vershalovskiy.com`: кнопки рендерятся как
`<a href="https://t.me/...">`, клик — прямой переход. `url_im` работает как редирект
**без какого-либо fallback-кода** — секция про PHP-обработчик `OnImConnectorMessageAdd`,
которая раньше была здесь «на всякий случай», убрана: она не нужна, поведение проверено.

## Шаг 5. Публикация в маркетплейс

1. Кабинет партнёра: https://vendors.bitrix24.ru/ → **Мои приложения → Добавить приложение**

2. **Технические характеристики** (форма добавления версии):
   - Тип: **Использовать REST API** / простое (статичное)
   - Файл: тот же `app.zip`
   - Права (scopes) — реальный набор по вызовам в `index.html` (см. Шаг 2):
     `im`, `imopenlines`, `placement`, `contact_center`
     (отдельного scope `imconnector` не существует — методы `imconnector.*` идут под `imopenlines`)
   - Разделы системы — то, с чем приложение реально работает: Открытые линии,
     Контакт-центр, Чат и уведомления, Встраивание приложений
   - **Минимальный тариф: «Базовый»**. Поле декларативное — в документации
     (`apidocs.bitrix24.ru/market/preparing-to-publish/how-to-add-app.html`) сказано
     только «укажите тариф, минимальный, на котором будет корректно работать
     приложение, ориентируясь на страницу сравнения тарифов»; технически ли оно
     блокирует установку на меньших тарифах или просто информирует пользователя —
     источник не уточняет. «Базовый» взят потому, что это первый тариф, на котором
     раздел «Маркетплейс приложений для Контакт-центра» вообще открыт — на
     «Бесплатном» он недоступен (живая проверка `bitrix24.ru/prices/compare_cloud_plans.php`:
     там крестик на «Бесплатном» и «по подписке» начиная с «Базового»). Само
     приложение (`imconnector.*`, `app.option.*`, `placement.bind`) не задействует
     ничего тарифно-зависимого — ограничение чисто платформенное, не от кода.
   - Опции встраивания:
     - **«Добавлять свою страницу и пункт в главном меню»** → ☑ включить — для
       статичных приложений это единственный постоянный способ снова открыть
       приложение (нет install-хука, см. CLAUDE.md). В описании стоит по-человечески
       упомянуть название пункта меню (например, «Свои ссылки на мессенджеры»), чтобы
       пользователь не терялся после установки — это совет для UX, а не формальное
       требование модерации (такого пункта нет ни в чеклисте, ни в общих требованиях
       apidocs.bitrix24.ru/market/preparing-to-publish/).
     - **«Встраиваться в BitrixMobile»** → ☐ выключить (мобильных доработок нет)

3. **Карточка решения**: краткое описание (суть в одной фразе, не дублировать
   название), полное описание (что делает / как настроить / куда за поддержкой),
   квадратная иконка без артефактов, скриншоты экранов настройки.

4. **Настройки продаж**: регион публикации, тип («бесплатно» / «по подписке»),
   данные контрагента — для платных решений требуется проверка и подписание
   соглашения с 1С-Битрикс.

5. Отправить на модерацию.

## История решений (чтобы не наступать на старые грабли повторно)

- **Вебхук для `imconnector.*` не работает** — методы коннекторов требуют контекста
  OAuth-приложения, через голый вебхук падают с `WRONG_AUTH_TYPE`. Тестовый вебхук
  `https://thefastb24.bitrix24.ru/rest/9/3bpdh8pl5ii0t15p/` остаётся в проекте только
  как референс на случай, если когда-нибудь понадобится event-обработчик (сейчас не нужен).
- **GitHub Pages больше не используется.** Раньше казалось, что для маркетплейса нужен
  внешний хостинг с одним стабильным URL (т.к. `PLACEMENT_HANDLER` открывается в iframe
  у каждого клиента на их собственном портале, и захардкоженный URL вида
  `thefastb24.bitrix24.ru/local/...` дал бы 404 у других клиентов). Решение оказалось проще:
  `HANDLER_URL = window.location.href.split('?')[0]` подставляет тот CDN-адрес, который
  Битрикс выдаёт **каждому порталу индивидуально** — внешний хостинг не нужен вообще.
- **Fallback через `OnImConnectorMessageAdd` не понадобился** — `url_im` сам по себе
  делает кнопку прямой ссылкой, проверено живым рендерингом виджета.
- **Страница почты — отдельный `mail.html` в том же `app.zip`.** Сначала пробовали
  встроить её 5-м экраном в `index.html` (по `?mailto=` параметру) — при открытии
  напрямую по `url_im` страница не рендерилась как HTML (показывался светящийся
  исходный код). Решение: отдельный файл `mail.html` в архиве рядом с `index.html`,
  ссылка формируется как `HANDLER_URL.replace(/[^/]*$/, 'mail.html') + '?to=' + email`
  — тот же CDN-адрес портала, без внешнего хостинга.
